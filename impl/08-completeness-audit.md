# 完整性审计（面向空白项目独立复刻）

目标：判断当前手册是否足以在任意技术栈（App/CLI/Web/服务端）复刻核心能力。

## 1. 结论

当前文档已覆盖主要链路，但若要求“仅凭手册在空白项目稳定复刻”，仍需明确以下强制项：

- 平台依赖边界（哪些能力是 Windows 专属，哪些可替换）
- 规范级约束（MUST/SHOULD）
- 状态机与错误码契约（统一跨语言实现）
- 去重与合并的一致性测试向量
- URL 提取失败与 authkey 失效的完整降级流程

本文件给出缺口清单与补齐标准。

## 2. 缺口清单（已补充/待遵循）

### 2.1 平台依赖

- 日志路径解析默认是 Windows 路径模型。
- 代理抓包方案依赖“系统代理 + 本地 MITM + 证书信任链”。
- 在移动端/沙盒环境可能无法改系统代理，需提供替代方案（手动粘贴 URL）。

### 2.2 协议规范缺口

- 之前文档更偏“源码说明”，缺少“实现规范语气（MUST/SHOULD）”。
- 不同语言实现容易在顺序、去重、类型转换上出现偏差。

### 2.3 测试向量缺口

- 需要固定输入样例来验证：
  - `mergeList` 在有/无 id 条件下的行为
  - 分页提前停止是否正确
  - 导入排序后时间线是否稳定

## 3. 复刻必须满足的能力分级

### P0（必须）

- authkey 获取（自动或手动 URL）
- query 参数清洗与预检
- 分页抓取与重试
- 增量合并与去重
- 本地持久化

### P1（强烈建议）

- UIGF 导入导出
- 结构化错误码
- 备份恢复
- 多账号支持

### P2（可选）

- Excel 导出
- 云端同步
- 高级策略分析（保底推断/规划）

## 4. 实现一致性红线

- 记录顺序必须统一为“旧 -> 新”。
- 合并算法必须先 ID 对齐，再窗口签名兜底。
- `id` 比较必须使用大整数语义（或字符串数值比较策略）。
- `time` 原始字符串不得覆盖写回格式化后的值。
- 导入必须“校验 -> 备份 -> 写入”，禁止反序。

## 5. 建议验收方式

- 使用同一份原始抓取 URL，在两种语言实现中跑完整流程。
- 对比导出 JSON 的：
  - 总条数
  - 每池条数
  - 每池最后一条 id
  - 每池五星位置序列
- 四项全部一致才视为“实现等价”。

# 提瓦特风格抽卡分析实施方案（适配当前 CLI）

更新时间：2026-04-14

## 1. 目标与范围

### 1.1 目标
在现有 `gwc` CLI 上实现“可生成提瓦特小助手风格分析结果”的能力，分两阶段：
1. 结构化分析结果（JSON）
2. 可分享卡片（PNG）

### 1.2 非目标
- 不做游戏内自动化操作
- 不修改抽卡数据，只做读取与统计

## 2. 当前基础能力评估

已具备：
- 抽卡数据获取与本地存储（`fetch`）
- 基础分析（按池统计 3/4/5 星，计算每次 5 星 pity）

短板：
- 缺少“卡池期次 + UP 元数据”
- 缺少“歪/定/定轨”判定逻辑
- 缺少卡片渲染模块

## 3. 总体架构

```text
输入数据
  ├─ 本地抽卡记录（已有）
  ├─ 卡池期次配置（新增）
  └─ 角色/武器元数据（新增）

分析引擎（新增 v2）
  ├─ 清洗与分池
  ├─ 事件归档（每次5星事件）
  ├─ 保底/歪定判定
  ├─ 指标计算
  └─ 运势等级映射

输出
  ├─ analyze --profile-json (JSON)
  └─ analyze --card-out report/...png (PNG)
```

## 4. 数据模型设计

## 4.1 新增文件
1. `src/data/banner-schedule.json`
- 字段：`poolType`, `startAt`, `endAt`, `up5`, `up4`, `version`, `phase`

2. `src/data/item-meta.json`
- 字段：`name`, `type`, `rarity`, `icon`

## 4.2 新增 TypeScript 类型
建议新增 `src/model/analysis.ts`：
- `BannerPeriod`
- `FiveStarEvent`
- `GachaProfile`
- `PoolTimeline`
- `LuckTier`

## 5. 核心算法设计

### 5.1 五星事件流构建
输入：按时间升序的池内抽卡记录
输出：`FiveStarEvent[]`

每个事件包含：
- `name`, `time`, `pool`, `pity`, `isUp`, `isGuaranteed`, `isOffBanner`, `periodId`

### 5.2 角色池歪/定判定
规则：
- 若本次 5 星不在当期 UP 列表 => `isOffBanner=true`
- 若上一次角色池 5 星为歪，则本次应判为大保底阶段
- `小保底不歪率`仅统计非大保底触发场景

### 5.3 武器池判定
规则：
- 需基于当期 UP 武器列表识别是否命中目标池内 UP
- 定轨点数（若无法从原始数据直接还原）可做“近似推断”并标注 `confidence`

### 5.4 指标计算
最小必做指标：
- `totalPulls`
- `fiveStarCount`
- `avgPity`
- `win5050Rate`
- `upCharacterCost`
- `upWeaponCost`
- `currentPityByPool`
- `timelineByPool`

## 6. 命令行接口设计

### 6.1 扩展 `analyze`
新增参数：
- `--profile`：输出高级画像（终端摘要）
- `--profile-json <file>`：输出完整 JSON
- `--card-out <file>`：渲染 PNG 卡片
- `--theme <name>`：卡片主题（默认 `tivaux-dark`）

示例：
```bash
gwc analyze --uid 123456789 --profile --profile-json report/gacha-profile.json
gwc analyze --uid 123456789 --card-out report/cards/summary.png
```

## 7. 卡片渲染设计

### 7.1 渲染方案
推荐：`HTML/CSS -> PNG`（Playwright 截图）
- 优点：视觉自由度高，最容易做出示例图风格
- 输出：1080x1920（移动端分享比例）

备选：`Satori/Resvg`
- 优点：无浏览器依赖
- 缺点：复杂布局成本更高

### 7.2 卡片内容分块
1. Header：运势等级、特征码、核心指标
2. Summary：限定/常驻五星统计 + 头像列表
3. Timeline：各池 5 星抽数条形图 + 歪/定标记
4. Footer：生成时间、数据覆盖区间

## 9. 里程碑计划

### M1（1-2 天）
- 接入 `banner-schedule.json`
- 输出 `--profile-json`
- 完成主要指标口径

验收：
- 对同一份数据重复计算，结果稳定一致

### M2（1-2 天）
- 终端摘要输出（欧皇度、各池进度）
- 增加单元测试和快照测试

验收：
- 测试覆盖“歪->大保底->定”典型链路

### M3（2-3 天）
- 卡片渲染与主题样式
- 输出 PNG 并写入 `report/cards`

验收：
- 生成图包含：总抽、均金、歪定率、池内条形图

## 10. 测试策略

1. 单元测试
- `pity` 计算
- 角色池 50/50 判定
- 武器池 UP 判定

2. 集成测试
- 用固定 JSON 样本执行 `analyze --profile-json`
- 断言关键字段值

3. 视觉回归测试
- 同样输入生成卡片，做像素差异阈值检测

## 11. 风险与规避

1. 卡池元数据不完整
- 规避：对未知期次标注 `unknown_period`，并降级只做基础统计

2. 武器定轨难以完全还原
- 规避：输出 `confidence` 字段，低置信度不展示“绝对结论”

3. 游戏机制后续调整
- 规避：将规则参数化（硬保底、定轨点数）写入配置
